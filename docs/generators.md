# Command Generation

Coconut Milk scans your `commands/` folder for annotated Lua files and generates typed wrappers for both the Lua runtime and the frontend.

## Annotations

Commands are annotated with LuaCATS-style comments plus a custom `@command` tag:

```lua
---@command hello
---@description Say hello to someone
---@param params { name?: string }
---@return string
local function hello(params, ctx)
  return "Hello, " .. (params.name or "user")
end

return { hello = hello }
```

### Annotation rules

- `---@command NAME` must be placed immediately above the function
- `---@param name TYPE` declares the parameter type
- `---@return TYPE` declares the return type
- `---@description` is optional but recommended
- Multiline descriptions are supported via continuation lines

### Supported types

| Type | Description |
|---|---|
| `string` | String |
| `number` | Number |
| `boolean` | Boolean |
| `table` | Generic table |
| `{ key: type }` | Object/map |
| `{ type[] }` | Array |
| `fun(x: type): type` | Function type |
| `type \| type` | Union |
| `type?` | Optional (same as `type \| nil`) |
| `any` | Any type |

### Multi-command files

A single file can export multiple commands:

```lua
---@command greet
---@param params { name: string }
---@return string
local function greet(params, ctx)
  return "Hello, " .. params.name
end

---@command goodbye
---@param params { name: string }
---@return string
local function goodbye(params, ctx)
  return "Goodbye, " .. params.name
end

return {
  greet = greet,
  goodbye = goodbye,
}
```

## Generated files

For each `.lua` file in `commands/`, the generator produces:

### `.g.lua` — Lua registration glue

```lua
-- Generated by Coconut Milk build pipeline.
-- Do not edit by hand.

local impl = require("hello")

---@type fun(ctx: CoconutContext)
local function register(ctx)
  ctx:bind("hello", impl.hello)
  return ctx
end

return register
```

The runtime auto-loads these `.g.lua` files at startup.

### `.d.ts` — TypeScript declarations

```typescript
/// <reference path="../scripts/coconut.d.ts" />

/** Say hello to someone */
declare function hello(params: { name?: string }): Promise<[string]>;
```

### `.g.js` — JavaScript wrappers

```javascript
// Auto-generated command wrappers. Do not edit.
// Uses coconut.call() for Lua command invocation.
// Plain JS with JSDoc — no build step required.
// @ts-check

/**
 * Say hello to someone
 * @param {{name?: string}} params
 * @returns {Promise<string>}
 */
export async function hello(params) {
  return coconut.call("hello", { params });
}
```

## Running the generator

```bash
# Single file
coconut generate commands/hello.lua --out-dir generated

# All files (from project root)
for f in commands/*.lua; do
  coconut generate "$f" --out-dir generated
done
```

---

## Build pipeline

The generator (`coconut generate`) parses Lua files using a simple state machine. It scans for `---@command` tags and extracts the function signature, parameters, and return types.

### Parser behavior

- The parser is intentionally simple — it only reads the annotation block and the function signature
- It does not parse the function body
- It supports both `function` and `local function` declarations
- It stops parsing at the closing `)` of the parameter list
- If a type annotation is missing, it falls back to `any`

---

## Frontend usage

Import the generated wrappers in your frontend code:

```typescript
import { hello } from "./generated/hello.g.js"

const result = await hello({ name: "Ada" })
```

Or use the generic `coconut.call()` API:

```javascript
const result = await coconut.call("hello", { name: "Ada" })
```

---

## Configuration

Set the output directory in `coconut.config.lua`:

```lua
return {
  generators = {
    output_dir = "generated"
  }
}
```

Or pass `--out-dir` to the generator directly.

## Type inference

When `---@param` is missing or incomplete:

- Parameter names are inferred from the function signature
- Missing types default to `any`
- The function signature is the fallback source of truth

```lua
---@command login
---@param params { username: string, password: string }
local function login(params, ctx)
  -- username and password are inferred from @param
end

---@command ping
local function ping(params, ctx)
  -- no @param — params and ctx are inferred as any
end
```
